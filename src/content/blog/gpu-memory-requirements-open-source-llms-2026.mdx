## Quick Answer

**The VRAM equation is simple: full-precision parameters × 2 bytes + 20% overhead = minimum training VRAM.** Llama 3 70B at 140GB requires multi-GPU. Mistral 7B at 14GB fits in RTX 4090. DeepSeek V3 at 320GB needs a cluster. Quantization (QLoRA, GPTQ, AWQ) cuts requirements by 60-75% with minimal quality loss for most tasks.

---

## Why VRAM Requirements Actually Matter

Choosing the right GPU for your model is not about preference—it is about math. A model that requires 140GB of VRAM cannot run on a 24GB RTX 4090, period. Understanding VRAM requirements prevents expensive mistakes: renting an A100 when an RTX 4090 would suffice, or buying RTX 4090s when your workload actually needs A100s.

This guide covers the most commonly deployed open-source models as of April 2026. Numbers are based on official model documentation, community benchmarks, and my own deployment experience.

**Key assumption:** We use fp16 (float16) as the baseline unless noted. Quantization examples use 4-bit unless noted.

## The VRAM Formula

Before listing specific models, understand the calculation:

### Inference VRAM

```
Model VRAM = Parameters × 2 bytes (fp16) × 1.2 (overhead)
```

70B model = 70,000,000,000 × 2 × 1.2 = **168GB**

The 1.2x overhead covers activation memory, KV cache, and runtime buffers. Actual requirements vary by implementation.

### Training VRAM

```
Training VRAM = Parameters × 2 bytes × 4 (optimizer + gradients + activations + model)
```

70B training = 70B × 2 × 4 = **560GB**

Training requires 3-4x inference VRAM due to optimizer states (Adam uses fp32, which is 4 bytes per parameter), gradients, and activation memory for backpropagation.

### QLoRA Fine-tuning VRAM

```
QLoRA VRAM = Parameters × 0.5 bytes (4-bit) × 1.2 + 4GB (adapter overhead)
```

70B QLoRA = 70B × 0.5 × 1.2 + 4GB = **~46GB**

QLoRA (Quantized Low-Rank Adaptation) enables 70B fine-tuning on single 80GB GPU by quantizing the base model to 4-bit and only training small adapter matrices in fp16.

## Multi-GPU Strategies for Large Models

When your model does not fit in single GPU, you need tensor parallelism or pipeline parallelism.

### Tensor Parallelism

Splits the model weights across GPUs along the tensor dimension. All GPUs work on each layer together. Requires high-bandwidth interconnects (NVLink).

- **Effective for:** Inference, training with large batch sizes
- **Communication overhead:** High (all GPUs must sync every layer)
- **NVLink required:** Yes for performance

### Pipeline Parallelism

Splits model layers across GPUs. GPU 1 processes layers 1-20, GPU 2 processes layers 21-40, etc. Latency from pipeline bubbles.

- **Effective for:** Training with small batch sizes
- **Communication overhead:** Lower than tensor parallelism
- **NVLink helpful but not required:** Data transfers via PCIe work with acceptable overhead

## Quantization: The Memory Reduction Multiplier

Quantization reduces weight precision to shrink VRAM requirements. The quality trade-off has improved dramatically with newer techniques.

### 4-bit Quantization (GPTQ, QLoRA)

- Reduces fp16 to 4-bit integers via calibration dataset
- 60-75% memory reduction with <5% quality loss on most benchmarks
- **Best for:** Production inference where quality preservation matters
- **Tools:** AutoGPTQ, GPTQ-for-LLaMa, llama.cpp

### 8-bit Quantization

- Reduces fp16 to 8-bit with mixed precision (some layers stay fp16)
- 40-50% memory reduction with <2% quality loss
- **Best for:** Cases where 4-bit quality loss is unacceptable but 16GB models are too small
- **Tools:** bitsandbytes 8-bit optimization

### 2-bit Quantization (Emerging)

- Emerging technique with active research in 2026
- 80% memory reduction but significant quality loss without special handling
- **Best for:** Research only, not production
- **Tools:** AWQ, QuaROT with calibration improvements

**1. What model quality do you need?**
